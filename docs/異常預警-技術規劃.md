# 暖暖「異常預警」功能 — 技術實作規劃

> **目標**：長輩出現健康異常 / 失聯時，自動通知子女（家人）
> **給 Cursor AI**：這是 build-ready spec，照 Phase 順序實作
> **核心價值**：把「被動記錄」升級成「主動守護」— 這是真正的護城河

---

## 🎯 為什麼這個功能值得做

子女最深的恐懼：「**爸媽出事了，我卻不知道**」。
現有所有銀髮 App、ChatGPT、健保 App 都**沒有「主動通知」**。
這是付費決策者（子女）願意掏錢的理由。

---

## 🏗️ 整體架構

```
每天早上 9:00（台灣）Vercel Cron 自動觸發
   │
   ▼
GET /api/cron/check-anomalies   (用 CRON_SECRET 驗證)
   │
   ├─ 用 service role 撈出「所有有 accepted 家人 + alerts 權限」的長輩
   │
   ├─ 對每位長輩跑 5 條偵測規則：
   │     1. 失聯（N 天沒任何記錄）   ← 最重要
   │     2. 血壓異常 / 連續升高
   │     3. 血糖異常
   │     4. 體重驟變
   │     5. 連續漏吃藥
   │
   ├─ 命中規則 → 查 alerts 表確認「24h 內沒發過同類型」（防重複）
   │
   └─ 寫入 alerts 表 + 用 Resend 寄 email 給子女
```

### Step 1.1 — 建 alerts 資料表

新檔案 `supabase/add-alerts.sql`：

```sql
-- 異常預警記錄
create table if not exists alerts (
  id uuid primary key default gen_random_uuid(),
  elder_id uuid references auth.users on delete cascade not null,  -- 長輩
  alert_type text not null check (alert_type in (
    'inactivity', 'blood_pressure', 'blood_glucose', 'weight_change', 'missed_medication'
  )),
  severity text not null default 'warning' check (severity in ('info', 'warning', 'critical')),
  title text not null,
  message text not null,
  metadata jsonb default '{}',           -- 存觸發時的數值（例如 systolic: 165）
  notified_family jsonb default '[]',    -- 通知了哪些家人 [{family_id, channel, sent_at}]
  resolved boolean default false,
  created_at timestamptz default now()
);

create index if not exists alerts_elder_type_idx
  on alerts (elder_id, alert_type, created_at desc);

alter table alerts enable row level security;

-- 長輩本人 + 被授權的家人都能看
create policy "elder reads own alerts"
  on alerts for select using (auth.uid() = elder_id);

create policy "family reads linked alerts"
  on alerts for select using (
    exists (
      select 1 from family_links
      where family_links.owner_id = alerts.elder_id
        and family_links.family_user_id = auth.uid()
        and family_links.status = 'accepted'
        and (family_links.permissions->>'alerts')::boolean = true
    )
  );
-- 寫入只由 service role（cron）做，不開 insert policy 給一般用戶
```

→ 到 Supabase SQL Editor 執行

### Step 1.2 — Resend 寄信工具

新檔案 `src/lib/email/send.ts`：

```ts
// 用 Resend API 直接寄信（跟 Supabase auth 寄信分開）
// 需要環境變數 RESEND_API_KEY（re_xxx）

interface SendEmailParams {
  to: string;
  subject: string;
  html: string;
}

export async function sendEmail({ to, subject, html }: SendEmailParams) {
  const apiKey = process.env.RESEND_API_KEY;
  if (!apiKey) {
    console.warn("[email] RESEND_API_KEY 未設定，跳過寄信");
    return { skipped: true };
  }

  const res = await fetch("https://api.resend.com/emails", {
    method: "POST",
    headers: {
      Authorization: `Bearer ${apiKey}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      from: "暖暖 <noreply@nuan55.com>",
      to,
      subject,
      html,
    }),
  });

  if (!res.ok) {
    const err = await res.text();
    console.error("[email] 寄信失敗:", err);
    return { error: err };
  }
  return await res.json();
}
```

> ⚠️ 要去 Resend 拿一把 API Key（https://resend.com/api-keys），加到 Vercel 環境變數 `RESEND_API_KEY`

### Step 1.3 — 偵測引擎（核心）

新檔案 `src/app/api/cron/check-anomalies/route.ts`：

```ts
import { NextRequest, NextResponse } from "next/server";
import { createSupabaseAdmin } from "@/lib/supabase/server";
import { sendEmail } from "@/lib/email/send";
import { buildAlertEmail } from "@/lib/email/templates";

export async function GET(req: NextRequest) {
  // 1. 驗證 cron secret（防止外人亂打）
  const auth = req.headers.get("authorization");
  if (auth !== `Bearer ${process.env.CRON_SECRET}`) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }

  const supabase = createSupabaseAdmin();
  const results: any[] = [];

  // 2. 撈出所有「有 accepted 家人 + alerts 權限」的長輩
  const { data: links } = await supabase
    .from("family_links")
    .select("owner_id, family_user_id, family_name, permissions")
    .eq("status", "accepted");

  // 以 owner_id 分組
  const elderMap = new Map<string, typeof links>();
  for (const link of links ?? []) {
    if (!(link.permissions as any)?.alerts) continue;
    if (!elderMap.has(link.owner_id)) elderMap.set(link.owner_id, []);
    elderMap.get(link.owner_id)!.push(link);
  }

  // 3. 對每位長輩跑規則
  for (const [elderId, familyLinks] of elderMap) {
    const alerts = await detectAnomalies(supabase, elderId);

    for (const alert of alerts) {
      // 防重複：24h 內發過同 type 就跳過
      const since = new Date(Date.now() - 24 * 3600 * 1000).toISOString();
      const { data: recent } = await supabase
        .from("alerts")
        .select("id")
        .eq("elder_id", elderId)
        .eq("alert_type", alert.type)
        .gte("created_at", since)
        .limit(1);
      if (recent && recent.length > 0) continue;

      // 寫入 alerts
      const { data: inserted } = await supabase
        .from("alerts")
        .insert({
          elder_id: elderId,
          alert_type: alert.type,
          severity: alert.severity,
          title: alert.title,
          message: alert.message,
          metadata: alert.metadata,
        })
        .select()
        .single();

      // 寄 email 給每個家人
      for (const fl of familyLinks) {
        if (!fl.family_user_id) continue;
        const { data: { user: familyUser } } =
          await supabase.auth.admin.getUserById(fl.family_user_id);
        if (!familyUser?.email) continue;

        // 取長輩名字
        const { data: elderProfile } = await supabase
          .from("profiles").select("display_name").eq("id", elderId).single();

        await sendEmail({
          to: familyUser.email,
          subject: `【暖暖】${elderProfile?.display_name ?? "您的家人"} ${alert.title}`,
          html: buildAlertEmail({
            familyName: fl.family_name,
            elderName: elderProfile?.display_name ?? "您的家人",
            alert,
          }),
        });
      }
      results.push({ elderId, type: alert.type });
    }
  }

  return NextResponse.json({ checked: elderMap.size, fired: results.length, results });
}

// ── 偵測規則 ──────────────────────────────
async function detectAnomalies(supabase: any, elderId: string) {
  const alerts: any[] = [];
  const now = Date.now();

  // 規則 1：失聯（3 天沒任何記錄）
  const threeDaysAgo = new Date(now - 3 * 24 * 3600 * 1000).toISOString();
  const [meals, metrics, convs] = await Promise.all([
    supabase.from("meals").select("id").eq("user_id", elderId).gte("eaten_at", threeDaysAgo).limit(1),
    supabase.from("health_metrics").select("id").eq("user_id", elderId).gte("measured_at", threeDaysAgo).limit(1),
    supabase.from("conversations").select("id").eq("user_id", elderId).gte("created_at", threeDaysAgo).limit(1),
  ]);
  const hasActivity =
    (meals.data?.length ?? 0) + (metrics.data?.length ?? 0) + (convs.data?.length ?? 0) > 0;
  if (!hasActivity) {
    alerts.push({
      type: "inactivity",
      severity: "warning",
      title: "已 3 天沒有使用暖暖",
      message: "您的家人已經連續 3 天沒有記錄飲食、量測或使用語音。建議主動關心一下他的狀況。",
      metadata: { days: 3 },
    });
  }

  // 規則 2：血壓異常（最近一筆）
  const { data: bp } = await supabase
    .from("health_metrics")
    .select("systolic, diastolic, measured_at")
    .eq("user_id", elderId)
    .eq("metric_type", "blood_pressure")
    .order("measured_at", { ascending: false })
    .limit(1);
  if (bp && bp[0]) {
    const { systolic, diastolic } = bp[0];
    if (systolic >= 160 || diastolic >= 100) {
      alerts.push({
        type: "blood_pressure",
        severity: "critical",
        title: "血壓偏高",
        message: `最近一次血壓 ${systolic}/${diastolic} mmHg，已超過警戒值（160/100）。建議盡快回診或聯繫醫師。`,
        metadata: { systolic, diastolic },
      });
    } else if (systolic <= 90 || diastolic <= 60) {
      alerts.push({
        type: "blood_pressure",
        severity: "warning",
        title: "血壓偏低",
        message: `最近一次血壓 ${systolic}/${diastolic} mmHg 偏低，請留意是否有頭暈、無力情形。`,
        metadata: { systolic, diastolic },
      });
    }
  }

  return alerts;
}
```

### Step 1.5 — Vercel Cron 設定

新檔案 `vercel.json`（專案根目錄）：

```json
{
  "crons": [
    {
      "path": "/api/cron/check-anomalies",
      "schedule": "0 1 * * *"
    }
  ]
}
```

> `0 1 * * *` = 每天 UTC 01:00 = **台灣早上 9:00**
> ⚠️ Vercel Cron 免費版**一天只能跑幾次**，每天 1 次完全夠用

**Cron 驗證**：Vercel 觸發 cron 時會自動帶 `Authorization: Bearer ${CRON_SECRET}`，所以要：
1. 設一個隨機字串當 `CRON_SECRET`
2. 加到 Vercel 環境變數
3. Vercel Cron 設定會自動帶這個 header（新版 Vercel 會用 `CRON_SECRET` 環境變數自動驗證）

## ⚠️ 重要注意事項

1. **醫療免責聲明**：警報文案要寫「建議」而非「診斷」，避免醫療責任。所有警報都要附「請諮詢專業醫師」。
2. **隱私**：只通知 status=accepted 且 permissions.alerts=true 的家人。長輩可隨時關閉。
3. **防騷擾**：24h 內同類型不重發，避免子女被轟炸。
4. **service role 安全**：cron 用 service role 繞過 RLS，務必用 CRON_SECRET 保護端點。
5. **閾值要保守**：寧可漏報也不要狂報假警報（會讓子女失去信任）。第一版閾值建議偏嚴格。
