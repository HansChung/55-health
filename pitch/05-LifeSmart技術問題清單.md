# 暖暖 × LifeSmart 整合技術問題清單

> **用途**：交給 LifeSmart 工程／開放平台窗口，確認串接細節與可行性
> **目標**：釐清「怎麼接、能拿到什麼資料、多即時、如何授權、資料放哪」
> **標記**：🔴＝必答（決定能否開工）　🟡＝重要　⚪＝後續

致 LifeSmart 技術團隊：

感謝 貴公司願意提供 API。暖暖（nuan55.com）是一套 55+ 銀髮健康守護軟體，已具備異常預警、家人通知與健康儀表板。我們希望承接 貴公司感測裝置的事件，做 AI 分析與家人通知。以下為串接前需要釐清的技術問題，煩請逐項回覆，謝謝。

---

## 一、開放平台 / 認證 Authentication

1. 🔴 是否有**開放平台 / 雲端 API 文件**？可否提供文件連結與開發者帳號申請方式？
2. 🔴 API 採用哪種認證？（OAuth 2.0／API Key／簽章？）授權流程為何？
3. 🔴 Token 的有效期與更新（refresh）機制？
4. 🟡 是正式（production）與測試（sandbox）分開的環境嗎？API base URL 各為何？
5. 🟡 申請接入需要哪些資料／審核流程？大約多久可開通？

---

## 二、裝置與資料 Devices & Data

6. 🔴 哪些**裝置 SKU** 提供資料 API？（人體存在偵測、跌倒／離床、SOS、智慧床墊、門磁、溫濕度、空氣品質…）
7. 🔴 每種裝置可取得**哪些欄位**？請提供資料格式範例（JSON）。
8. 🔴 各數值的**單位、時間戳格式與時區**為何？（例如溫度 °C、時間 ISO8601 UTC？）
9. 🟡 裝置與「使用者帳號」如何對應？一位長輩名下多個裝置如何識別？
10. 🟡 是否支援多租戶／多家庭的資料隔離？

---

## 三、即時性與推播 Real-time & Push

11. 🔴 取得事件是用 **Webhook 推播**，還是要我們**輪詢（polling）**？
12. 🔴 若有 Webhook：可否推送到我們指定的 HTTPS 端點？推播格式為何？
13. 🔴 **緊急事件（跌倒／SOS）的延遲**大約多少秒？（這對救援最關鍵）
14. 🟡 若是輪詢：建議頻率？有無最小間隔限制？
15. 🟡 事件是否保證**順序與不漏送**？是否有重送（retry）與去重（event id）機制？

---

## 四、用量與限制 Quota & Limits

16. 🟡 API 的**呼叫頻率限制（rate limit）**為何？
17. ⚪ 單帳號可綁定的裝置數上限？
18. ⚪ 歷史資料可回查多久？是否提供歷史查詢 API？

---

## 五、安全 Security

19. 🔴 Webhook 是否提供**簽章驗證**（讓我們確認事件真的來自 LifeSmart）？
20. 🟡 是否支援 IP 白名單 / mTLS？
21. 🟡 個人健康資料傳輸是否全程 HTTPS／加密？

---

## 六、資料落地與合規 Data Residency & Compliance ⚠️

22. 🔴 API 與資料伺服器**機房位置**在哪？（中國／海外節點？）
23. 🔴 台灣使用者的資料能否**落地台灣或海外節點**？（涉及台灣個資法，是合作能否上線的關鍵）
24. 🟡 資料的擁有權與授權範圍？我們可否將事件存入自己的資料庫做分析？
25. ⚪ 是否有資料處理協議（DPA）／隱私條款可簽署？

---

## 七、測試與支援 Testing & Support

26. 🟡 可否提供**測試帳號 + 測試裝置**（或模擬器），讓我們先做 POC？
27. 🟡 是否有技術窗口／開發者社群／工單支援？
28. ⚪ 是否有 SDK（如 Node.js / 後端）可直接使用，還是純 REST？

---

## 八、商務（技術相關）Commercial

29. 🟡 API 介接是否**收費**？怎麼計（按裝置／呼叫量／月費）？
30. ⚪ 硬體由誰供應、安裝？是否有給合作夥伴的價格？

---

## ⭐ 我們最想先確認的三件事（若時間有限，請先回這三題）

1. **是否有開放 API、用什麼認證**（題 1–3）
2. **緊急事件（跌倒／SOS）怎麼即時拿到、延遲多少**（題 11–13）
3. **台灣用戶資料能否落地台灣／海外**（題 22–23）

確認以上三點，我們即可評估開工時程與 POC 範圍。再次感謝！
